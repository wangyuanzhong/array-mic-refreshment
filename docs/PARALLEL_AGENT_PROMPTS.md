# Parallel Agent Prompts（Wake Word + PTT）

> 目标：并行推进 UI 与核心链路，减少冲突，最后由主 agent 合并与回归测试。

## Agent A（UI + Settings）

请在分支 `cursor/wakeword-ui-settings-91a6` 上完成以下任务：

1. 在 `SettingsForm` 增加“触发模式”下拉：
   - `PTT`
   - `唤醒词`
2. 当触发模式为“唤醒词”时，显示并保存“唤醒词文本”输入框；PTT 模式下可隐藏或禁用该输入框。
3. 调整设置窗尺寸与布局，确保新增控件在 100% 缩放下无需裁切且可滚动。
4. 持久化配置：
   - 在 `AppSettings` 增加触发模式与唤醒词字段（带合理默认值，兼容旧配置）。
   - 更新 `SettingsCopier` 和相关加载/保存逻辑。
5. 不要实现唤醒词识别算法，只做 UI + 配置面。
6. 保持现有 PTT 行为不变。

交付要求：
- 更新/新增针对设置持久化的单元测试（如果已有测试路径可复用）。
- 提交、推送分支并创建/更新 PR。
- 在 PR 描述中列出修改文件和 UI 交互变化。

---

## Agent B（Core 架构 + 监控定时器）

请在分支 `cursor/wakeword-core-arch-91a6` 上完成以下任务：

1. 设计并落地触发模式路由骨架（不必一次到位接入完整模型）：
   - `PTT`：沿用现有路径；
   - `唤醒词`：预留/接入常驻监听入口（可先用 stub detector），保证与 `VoicePipeline` 的连接点清晰。
2. 保持与现有 `TrayApplicationContext` 的集成清晰，确保模式切换不会影响 PTT 回归。
3. 实现一个“定时轮询监控状态”的工具脚本（建议 Python）：
   - 输入两个或多个分支名；
   - 每 N 秒轮询一次；
   - 输出分支提交进度（ahead/behind、最新 commit、变更文件数）；
   - 可选输出 PR/CI 状态（若有 `gh`）。
4. 为监控脚本提供命令示例。

交付要求：
- 提供至少 1 条脚本运行验证命令（one-shot 轮询即可）。
- 提交、推送分支并创建/更新 PR。
- 在 PR 描述中标注：哪些是骨架、哪些是可直接用于生产的逻辑。

---

## 合并前约束（两位 agent 都必须遵守）

1. 不改动彼此负责区域之外的大段代码，降低冲突。
2. 不删除现有功能，不更改对外行为（除新增功能开关外）。
3. 所有新增配置项必须有默认值，旧配置文件可直接加载。
4. 保持日志可读，错误信息明确。
